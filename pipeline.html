Tóm tắt thiết kế: Bước 1–2 là tầng dữ liệu và quyết định — không dùng LLM. Bước 3 (Planner) tạo blueprint có ambiguity_risk để detect lỗi ngôn ngữ sớm. Bước 4 (Generator) sinh câu hỏi hoàn chỉnh với domain rotation thực sự ngẫu nhiên. Bước 5 là tầng thay thế Auto Solver — gồm Linguistic Clarity Check (LLM nhỏ) và Ground Truth Assertion (rule-based). 01
BƯỚC 01 Skill Taxonomy DB
Nguồn tri thức Không LLM · $0
▼
💡 Vai trò: "Từ điển kiến thức SQL" — định nghĩa tất cả topic, độ khó, lỗi phổ biến, pattern câu hỏi tốt. Không sinh câu hỏi, chỉ cung cấp nền tảng để các bước sau đọc và bám vào.
⭐ Cold Start Protocol (mới): Trước khi pipeline chạy lần đầu, admin cần build taxonomy thủ công từ syllabus PTIT + đề LearnSQL cũ, và viết tay 3–5 câu mẫu chất lượng cao cho mỗi topic. Các câu này tag source: "human_seed" và có trọng số few-shot cao hơn câu AI sinh. Chỉ chuyển sang semi-dynamic sau khi có đủ 4 tuần data thực.
📥 Input
Syllabus PTIT, đề thi LearnSQL cũ, community tips, analytics (khi có)
📤 Output
taxonomy.json — cây topic đầy đủ với common_mistakes, difficulty_signals, good_question_patterns
Cấu trúc taxonomy entry:
{
  "JOIN": {
    "level": 2,
    "prerequisites": ["SELECT", "WHERE", "alias"],
    "subtypes": ["INNER", "LEFT", "RIGHT", "SELF"],
    "common_mistakes": [
      "quên ON condition → cartesian product",
      "nhầm LEFT vs INNER khi có NULL"
    ],
    "difficulty_signals": {
      "easy":   "2 bảng, INNER JOIN, output rõ ràng",
      "medium": "3 bảng, LEFT JOIN, có NULL",
      "hard":   "self join hoặc nhiều điều kiện ON"
    },
    "few_shot_seeds": ["q_seed_join_001", "q_seed_join_002"] // human_seed refs
  }
}
Giai đoạn cập nhật:
  • Giai đoạn 1 (tuần 1–4): Static hoàn toàn. Admin build thủ công, chỉ thay đổi khi admin quyết định.
  • Giai đoạn 2+ (sau 4 tuần): Mỗi thứ Hai, analytics đề xuất update weight. Admin approve trước khi apply.
02
BƯỚC 02 Skill Target Selector
Rule-based Không LLM · $0 Nâng cấp
▼
💡 Vai trò: "Lập kế hoạch tuần" — quyết định tuần này sinh bao nhiêu câu, topic nào, độ khó ra sao. Nâng cấp so với pipeline gốc: thay vì chỉ nhìn pass rate, giờ chẩn đoán được nguyên nhân tại sao pass rate thấp và xử lý khác nhau.
⭐ Chẩn đoán nguyên nhân (mới): Pass rate thấp có thể do câu quá khó (cần sinh câu dễ hơn) hoặc do sinh viên chưa hiểu concept (cần sinh câu có tutorial mode). Hai trường hợp này cần xử lý ngược nhau — pipeline gốc không phân biệt được. Chỉ bật chẩn đoán khi có đủ sample_size >= 30 để tránh quyết định sai từ data ít.
📥 Input
taxonomy.json, analytics tuần trước (pass rate, avg_attempts, hint_usage, avg_time), config admin
📤 Output
target_spec — list topic + difficulty + số lượng + diagnosis + tutorial_mode flag
Logic chẩn đoán:
def diagnose_topic(stats):
    # Chỉ chạy khi đủ data — tránh quyết định sai từ sample nhỏ
    if stats['sample_size'] < 30:
        return 'insufficient_data'  # → dùng preset syllabus

    if stats['pass_rate'] < 0.6:
        if stats['avg_attempts'] > 5 and stats['avg_time'] > 600:
            return 'too_hard'      # → sinh câu dễ hơn cùng topic
        elif stats['hint_usage'] > 0.7 and stats['avg_attempts'] <= 2:
            return 'concept_gap'   # → easy + tutorial_mode: true
        elif stats['avg_attempts'] > 3 and stats['hint_usage'] < 0.3:
            return 'ambiguous'     # → flag câu cũ, tạm dừng sinh thêm

    return 'healthy'             # → giữ nguyên phân phối
Hành động theo diagnosis:
  • too_hard: Sinh câu difficulty: "easy" cùng topic trong tuần tới.
  • concept_gap: Sinh câu easy + bật tutorial_mode: true — câu hỏi có nhiều hint hơn, gợi ý từng bước.
  • ambiguous: Flag câu hỏi cũ để admin review, tạm dừng sinh thêm topic đó tuần này.
  • Admin override luôn thắng: Pipeline phục vụ người dùng, không phải ngược lại.

BƯỚC 03 Question Planner
LLM #1 Gemini Free +ambiguity_risk
▼
💡 Vai trò: Tạo "kịch bản bẫy sư phạm" trước khi viết đề. Điểm khác biệt lớn nhất so với các pipeline thông thường — thay vì để LLM sáng tác tự do, bắt nó lên kế hoạch trước. Blueprint là checkpoint: nếu sai thì reject trước khi tốn call Generator.
⭐ Field ambiguity_risk (mới): Planner tự đánh giá rủi ro mơ hồ ngay từ lúc lên blueprint — phát hiện sớm hơn 2 bước so với pipeline gốc. Nếu không rỗng, Generator bắt buộc phải viết question_text rõ ràng theo required_clarification. Giảm đáng kể tải cho Bước 5.
📥 Input
target_spec từ bước 2, taxonomy entry của topic (chỉ inject phần liên quan — tiết kiệm token), few-shot minimal (1 example ~150 tokens)
📤 Output
question_blueprint — ý định + schema hint + trap strategy + ambiguity_risk
Blueprint có ambiguity_risk:
{
  "intent": "Test phân biệt INNER vs LEFT JOIN khi có NULL",
  "target_skill": ["LEFT JOIN", "NULL handling"],
  "difficulty": "medium",
  "schema_hint": {
    "tables": ["orders", "customers"],
    "key_design": "có customer chưa có order nào (NULL side)"
  },
  "question_strategy": {
    "trap": "dùng INNER JOIN sẽ bỏ mất khách chưa có đơn",
    "expected_concept": "LEFT JOIN + COUNT + COALESCE"
  },
  "ambiguity_risk": {           // ⭐ field mới
    "potential_misreads": [],   // rỗng = không có rủi ro
    "required_clarification": ""
  }
}
⚙️ Config:
  • temperature=0.3 — cần blueprint nhất quán, không cần creative.
  • Token tối ưu: Chỉ inject taxonomy entry của topic đang sinh, few-shot rút gọn còn ~150 tokens thay vì ~600 tokens của pipeline gốc.

BƯỚC 04 Question Generator
LLM #2 Gemini Free +Domain Rotation

💡 Vai trò: AI chính thức "chắp bút" — bám sát blueprint để viết đề hoàn chỉnh bằng tiếng Việt, schema SQL, seed data, answer query. Không sáng tác tự do. Nếu blueprint có ambiguity_risk không rỗng → bắt buộc follow required_clarification.
⭐ Domain Rotation thực sự ngẫu nhiên (mới): Pipeline gốc xoay domain theo lịch cố định — sinh viên có thể đoán. Mỗi câu trong tuần random 1 trong 5 domain → pool đa dạng domain ngay trong cùng tuần.

📤 Output

05
BƯỚC 05 Clarity Check + Ground Truth Assertion
LLM #3 nhỏ Rule-based Gemini Free Thay Auto Solver
▼
💡 Vai trò: Thay thế Auto Solver của pipeline gốc. Gồm 2 lớp chạy song song — Lớp A bắt lỗi ngôn ngữ/ngữ nghĩa (LLM), Lớp B bắt lỗi logic SQL (rule-based). Hai lớp bổ sung cho nhau, không cái nào claim 100% nhưng cộng lại cover được đa số case thực tế.
Lớp A — Linguistic Clarity Check

LLM đọc câu hỏi, liệt kê các cách hiểu khác nhau mà sinh viên có thể có. Task hẹp → dùng model nhỏ được.

Prompt:
"Đọc câu hỏi SQL sau.
Liệt kê TẤT CẢ các cách hiểu khác
nhau mà một sinh viên có thể có.
Nếu chỉ có 1 cách hiểu → trả về [].
Respond ONLY với JSON array."

[] → pass ✓
["cách hiểu 1",...] → rewrite question_text

Khi fail: gửi danh sách cách hiểu về Generator để rewrite question_text — không reject toàn bộ, chỉ rewrite phần câu hỏi.

Lớp B — Ground Truth Assertion

Rule-based, $0, không dùng LLM. Chỉ implement 2 test thực tế nhất:

  • Negation test: Thêm 1 row KHÔNG thỏa mãn điều kiện → verify không xuất hiện trong output. Bắt sai logic WHERE/HAVING/JOIN.
  • NULL test: Nếu schema có nullable column → thêm row NULL → verify handle đúng. Bắt lỗi quên COALESCE, sai JOIN type.

Không implement boundary test phức tạp — 70% coverage với 30% effort.

Tại sao không giữ Auto Solver gốc:
  • Auto Solver dùng LLM nặng để giải lại toàn bộ câu hỏi — tốn nhiều token và gọi model lớn.
  • Linguistic Clarity Check làm đúng việc của Solver (bắt lỗi ngữ nghĩa) với task hẹp hơn nhiều → model nhỏ hơn đủ dùng → chi phí thấp hơn.
  • Ground Truth Assertion bắt thêm lỗi logic mà Solver gốc không bắt được (khi cả 2 LLM cùng sai).
Flow xử lý khi fail:
# Lớp A fail (ambiguous)
→ Gửi danh sách misreads về Generator
→ Rewrite question_text theo required_clarification
→ Re-check Lớp A (tối đa 1 lần)

# Lớp B fail (logic error)
→ Retry từ Generator với cùng blueprint (attempt 1)
→ Nếu vẫn fail → escalate lên Bước 7 retry policy

BƯỚC 06 Execution Engine Verify
Deterministic DB Không LLM · $0
▼
💡 Vai trò: "Chốt chặn cơ học" — chạy answer_query thật trên MySQL sandbox. Không có LLM ở tầng này → kết quả binary: pass hoặc fail, không có "có vẻ đúng". Safety net đáng tin nhất trong toàn pipeline.
📥 Input
schema_sql, seed_data_sql, answer_query, expected_output
📤 Output
verify_result — pass/fail + actual_output + flags
Checklist verify:
  • Syntax & runtime: Answer query có chạy được không, có lỗi cú pháp không.
  • Performance flag: Query chạy > 3s trên seed data nhỏ → không reject, ghi performance_issue vào metadata để reviewer biết.
  • Output không rỗng: Expected output phải có ít nhất 1 row.
  • Hash-based comparison: Chuẩn hóa data types, sort rows nếu câu không yêu cầu ORDER BY, so sánh set.
  • FK consistency: Seed data vi phạm referential integrity → reject ngay.
  • DDL handling: Với CREATE/ALTER/DROP → so sánh schema structure sau execution thay vì row output.
Output comparison logic:
def compare_output(actual, expected, requires_order=False):
    actual_norm   = normalize(actual)    # strip, lowercase, round float
    expected_norm = normalize(expected)

    if not requires_order:
        actual_norm   = sorted(actual_norm)
        expected_norm = sorted(expected_norm)

    actual_hashes   = [hash_row(r) for r in actual_norm]
    expected_hashes = [hash_row(r) for r in expected_norm]

    return actual_hashes == expected_hashes

BƯỚC 07 Output & Metadata + Retry Policy
Storage Không LLM · $0 +Retry Escalation
▼
💡 Vai trò: Đóng gói câu hỏi đã validated vào question bank ở trạng thái pending_review. Nâng cấp quan trọng: thêm retry escalation 3 cấp với archive fallback — đảm bảo đủ 100 câu mỗi tuần dù generation fail.
⭐ Retry Escalation 3 cấp (mới): Pipeline gốc chỉ "retry 3 lần" mà không nói sau đó thì sao — pool tuần đó có thể bị thiếu câu. Pipeline mới có escalation rõ ràng và archive fallback để đảm bảo SLA 100 câu/tuần.
Retry escalation:
Attempt 1
Generator chạy lại với cùng blueprint. Dùng khi: execution error, clarity fail sau rewrite.
Attempt 2
Generator chạy lại với blueprint đơn giản hóa — difficulty giảm 1 bậc, bỏ bớt trap. Dùng khi fail 2 lần liên tiếp.
Attempt 3
Ghi log generation_failed, skip topic này tuần này. Tự động lấy câu từ archive pool (câu cũ đã approved nhưng chưa active) để đảm bảo đủ 100 câu/tuần.
